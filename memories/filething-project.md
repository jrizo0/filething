**filething**: app de línea de comandos (CLI) para sincronizar carpetas/archivos entre dispositivos (Mac, Windows, VPS Linux), tipo Dropbox pero para developers, súper simple de iniciar y loguear. Es un producto SaaS. Working dir `/Users/jrizo/filething` (vacío al 2026-06-23, proyecto desde cero). Diseño en curso vía entrevista /grill-me.

Decisiones fijadas (al 2026-06-23):
- **Motor propio**, construido desde cero (lógica de sync en el cliente).
- **Sync continuo bidireccional** estilo Dropbox, en background (necesita file-watcher + ignores fuertes).
- **Unidad de sync = Spaces con nombre — RESUELTO 2026-06-24 (F1):** un Space = UNA sola raíz (un árbol del FS), mapeo UNO A UNO con una carpeta local por Device (la ruta puede variar por Device). Puedes tener tantos Spaces como quieras; cada Device elige cuáles sincroniza (control granular = elegir qué Spaces, por Device; sync por Space ENTERO, sin sync parcial de subcarpetas en v1). NO multi-raíz (varios árboles bajo un Space) ni el modelo monolítico de una sola carpeta gigante. Agrupar varias carpetas en un comando = comodidad de CLI post-MVP ("workspace"), NO concepto de dominio. (Término correcto: "Device", no "equipo" — "equipo" se confunde con el tier Team.)
- **Negocio = "cobrar por conectar"** (modelo Tailscale): vendes la coordinación por equipo/seat; los archivos van directo al almacenamiento del usuario y, en modo trae-tu-propio, nunca tocan la infra de Julian (costo ~$0 por esos usuarios).
- **Login con cuenta vía navegador** (GitHub/Google/email); `filething login` abre el navegador. Para VPS sin navegador, flujo de código.
- **Almacenamiento pluggable** = una "copia siempre encendida" contra la que sincroniza cada equipo. Solo DOS opciones: bucket rentado (R2) o el VPS del usuario. NO se soporta "trae tu propio bucket" (el usuario lo descartó el 2026-06-24 como otra feature que mantener). Todo se ve como un bucket compatible con S3 por dentro.
- **Para usuarios con VPS propio:** usan su VPS como su servidor de almacenamiento, pero el servicio de Julian SIGUE en el medio coordinando (login, emparejar, avisos) — estilo Tailscale. Julian sigue ganando; el user tiene storage casi gratis y privado.

Investigación de costos (verificada contra páginas oficiales 2026-06-23):
- **R2 gana** en costo: egress (bajar) gratis, ~$1.34/usuario/mes. **S3** ~6x más caro (egress $0.09/GB). **Convex** ~9x más caro por byte y malo para archivos grandes (límites reales verificados jun 2026: 16 MiB de args / 1 MiB por documento / 16 MiB leídos-escritos por transacción / mutation 1 s de CPU, actions hasta 10 min; por eso es "cerebro", NO almacén — el dato previo de "timeout 2 min / 20 MB" era impreciso).
- **Convex no compite como almacén sino como "cerebro"**: backend completo con tiempo-real (suscripciones reactivas por WebSocket) + base de datos, ya hechos. Componente oficial `get-convex/r2` para combinar Convex (cerebro) + R2 (bytes).
- **Arquitectura nube = híbrida: R2 para los bytes + Convex como cerebro.**

Stack fijado:
- **Cerebro = Convex gestionado** (tiempo-real + base de datos ya hechos; solo maneja metadata, no bytes; diseñar el cliente para poder migrar a un control plane propio si crece mucho). Convex tiene cliente oficial para TS, Python, Rust, Swift, Kotlin — NO Go/Java.
- **CLI = Rust** (cliente oficial de Convex con tiempo-real + un solo binario + fuerte para el motor de sync).
- **Monorepo** (polyglot Rust + TypeScript). Estructura propuesta: `apps/cli` (Rust), `apps/web` (Next.js, dashboard+billing), `apps/marketing`, `packages/backend` (Convex TS), `packages/shared`. Herramientas: pnpm workspaces + Turborepo (TS) y Cargo (Rust).
- **Web = Next.js** (dashboard + billing con Stripe + sitio de marketing; mejor SEO y ecosistema maduro).

Motor/comportamiento fijado:
- **Cifrado del lado del cliente, modelo CUSTODIA (escrow) — RESUELTO 2026-06-24 (F5/F6/F7).** El CONTENIDO se cifra (cada Block con una llave al azar, envuelta por la Space key); la METADATA (nombres/rutas/árbol) va en CLARO — al usuario NO le importa que filething vea nombres, solo el contenido. La Space key la CUSTODIA la cuenta (envuelta, recuperable vía login Google/GitHub), así que perder TODOS los devices NO deja sin acceso. Trade-off ACEPTADO por el usuario: filething-la-empresa técnicamente PODRÍA descifrar (es actor de confianza, como Google/GitHub); "cifrado" protege contra el proveedor de storage/robo del vault/atacantes externos, NO contra filething mismo ni una orden judicial. Se QUITA la promesa de "true zero-knowledge". Modo "ni nosotros podemos leerlo" (con frase de recuperación, asumiendo riesgo de lockout) = opción avanzada FUTURA, no MVP, no promesa. Auth SIMPLE: login Google/GitHub = identidad; 2FA opcional, NO obligatorio. Revocación de device perdido: rotar la Space key (nueva época) + cortar su acceso al vault (en serve los bytes van directo device→VPS, cortar solo el Coordinator no basta); honesto: protege lo FUTURO, no lo ya copiado.
- **Sync por bloques (delta) — RESUELTO 2026-06-24 (F6), REFINADO al aterrizar el formato (ver docs/format.md):** direccionamiento por hash del CIFRADO (ciphertext), NO del texto claro; el hash del claro ("pcid") es la llave de dedup, scope CUENTA, nunca cross-cuenta. Secreto de chunking del CDC = aleatorio por-Space (NO derivado de la Space key). CORRECCIÓN clave: la data key por-Block NO es puramente aleatoria (eso ROMPE el dedup cross-device bajo cifrado: dos devices cifrarían el mismo claro a objetos distintos). Es DETERMINISTA por contenido pero con secreto por-cuenta: data_key = KDF("dedup secret" aleatorio de 256 bits por-CUENTA, pcid); nonce idem. Así: mismo archivo en tus devices → mismo objeto → dedup real; otra cuenta → dedup secret distinto → ciphertext distinto → ni dedup ni leak cross-cuenta (NO es cifrado convergente puro). La data key se envuelve además con la Space key (en un sidecar keys/<cid>, fuera del cuerpo hasheado) para rotación barata = re-envolver sin re-cifrar ni renombrar. cid = BLAKE3(nonce || payload), excluye la data key envuelta. Un objeto por Block (borrar gratis en R2). En MVP el cifrado va OFF (bloques en claro, alg=0, cid==pcid) pero el formato ya reserva todo. (Mitigar canal lateral de tamaños de chunk de FastCDC: asignación aleatoria de chunks a packs, estilo restic 0.18.0 — packing es post-MVP.)
- **Sin ignores por defecto + cero interferencia**: el `.filethingignore` (por espacio) existe pero viene vacío; no hay avisos ni sugerencias. El usuario quiere sincronizar TODO lo que NO excluya explícitamente. MATIZ — RESUELTO 2026-06-24 (F2): distinguir dos cosas que el lenguaje confundía — (a) Ignore file = exclusión ELEGIDA por el usuario, vacío por defecto ("filething nunca descarta lo que NO elegiste descartar"); (b) política automática del motor sobre "Derived path" (node_modules/builds/venv NO viajan byte-a-byte: se sincroniza fuente+lock y se deja la ruta lista para REGENERAR con un comando confirmado — el motor NUNCA auto-ejecuta; .git SÍ viaja, es portable). Nuevo término de dominio = "Derived path" (en CONTEXT.md); la regla detallada del motor irá a un ADR. Implica motor eficiente con cientos de miles de archivos y alto churn. (Gotcha de binarios nativos por SO → RESUELTO, ver decisión de Binarios/artefactos, opción A híbrido por gestor.)

Binarios/artefactos por SO — RESUELTO 2026-06-24 (opción A: híbrido por gestor detectado). El default NO es "rutas por SO" sino "estrategia según el TIPO de artefacto", detectado por lockfile: pnpm/Yarn-PnP → NO sincronizar node_modules (es un grafo de symlinks/hardlinks a un store global FUERA del Espacio; copiarlo por-SO da symlinks rotos), sincronizar fuente+lock y reconstruir local; venv → siempre regen (guarda rutas absolutas); node_modules plano (npm/yarn clásico) → variante por SO viable; target/ de Rust y .next/ → no sincronizar por defecto, por-SO solo opt-in; caché de Go → compartido; .git → compartido normal (es portable; solo los hooks no lo son). "Variante por SO" queda solo como opt-in consciente. Default al regenerar = SOLO SUGERIR + confirmar, NUNCA auto-ejecutar (correr install = ejecutar código arbitrario); en serve mode/VPS nunca sin opt-in explícito. Symlinks en el motor: relativos dentro del Espacio se preservan; los de target absoluto o que salen del Espacio se materializan como copia o se marcan local-only. (Corrige el plan previo de "rutas por SO por defecto", que se rompe justo en pnpm — el gestor del propio stack.)

Más decisiones fijadas:
- **Manejo de conflictos = quedarse con ambas** (copia de conflicto); nunca perder datos en silencio; + historial. CRITERIO TÉCNICO — RESUELTO 2026-06-24 (F8): detección CAUSAL, nunca por reloj de pared. Cada Revision tiene UN padre (la Revision base del Device); historial en LÍNEA RECTA por Space; el Coordinator es árbitro del orden vía un puntero diminuto ("Space head") con compare-and-swap (no choca con el límite de 1 MiB de Convex: las páginas del Manifest se escriben antes, inmutables). Dos niveles: (1) "me quedé atrás" a nivel de Space = el commit hace CAS con su base; si base != head, el Device reconcilia (re-pull + re-diff), NO es conflicto aún; (2) conflicto real = por ARCHIVO: un mismo path cambió en AMBOS lados respecto a la base común → copia de conflicto; si cambió en un solo lado → fast-forward limpio (editar dos archivos distintos NO genera conflicto). Borrar-vs-editar el mismo archivo → gana la EDICIÓN, el borrado a historial. Re-scan que ve un archivo en el índice pero ausente en disco → tratar como borrado (se propaga), seguro porque hay historial. Descartados: vectores de versión (reinventa detección P2P innecesaria + bug conocido de Syncthing) y DAG estilo git (complejidad que el producto esconde). Rename-tracking y leases multi-persona = post-MVP.
- **CLI estilo git**: `filething login` (1 vez por máquina), `init` (carpeta → espacio), `clone <espacio> <ruta>` (traer en otra máquina, ruta configurable por equipo), `status`, `ls`. Modelo mental que los devs ya conocen.
- **Auth = Better Auth** (open-source para TS, gratis, control total, soporte Convex).
- **Billing = Polar** (merchant of record: maneja impuestos/IVA globales; enfocado en devs; soporta cobro por uso para el add-on de almacenamiento; integra con Better Auth). Stripe descartado por la carga fiscal para un fundador solo.

Decisiones resueltas el 2026-06-24 (entrevista, con investigación verificada vía workflow):

- **DAEMON:** un solo daemon multi-espacio (una conexión al change feed = encaja con Seat=Device; índice de bloques y dedup compartidos; supervisión interna por espacio). DOS perfiles explícitos del mismo binario: "desktop" (nivel-usuario, default, se registra en `init`, sin sudo) y `filething serve` (nivel-sistema, para VPS/Mac headless/self-hosted vault; mismo verbo que el vault serve-mode). NO auto-escalar a sistema por detectar SSH (solo SUGERIR `serve` si el modo usuario no sobrevive boot/logout). Canal CLI↔daemon = socket local (Unix socket/named pipe), no HTTP+token. Hallazgo clave: el nivel-usuario NO arranca headless en NINGÚN SO sin escalar (Linux: `systemd --user` muere al cerrar sesión salvo `loginctl enable-linger`; macOS: LaunchAgent necesita login GUI → usar LaunchDaemon root; Windows: tarea "al iniciar sesión" no dispara sin login → Windows Service). Clave en modo serve: single-tenant por host, login interactivo 1 vez por SSH, archivo 0600 de root (no keychain). Auto-update: post-MVP; delegar a gestor (brew/apt/msi) y auto-descarga firmada solo en instalaciones curl|sh.

- **HISTORIAL + GC:** motor = generation-marking (cada Block con last_seen_gen; el Coordinador marca en lotes <16k docs los Blocks alcanzables desde Manifests vivos; se borra lo no marcado tras gracia). Esquiva el refcount "una fila por hash" que revienta los límites de Convex. Borrado reversible con tombstone + período de gracia (estilo Borg 2.0). Formato del Vault = UN OBJETO POR BLOQUE (borrar = DeleteObject, gratis en R2; único costo = LIST/marcado, Class A). El Coordinador marca/decide (solo metadatos); el barrido físico lo hace un device siempre-encendido (gestionado: worker programado; self-hosted: el `serve` del VPS con un Bloom filter de "hashes vivos" que nunca borra un vivo). Retención por VENTANAS DE TIEMPO (no por conteo de Revisions), colapsando el churn a una Revision representativa por rato. Regla fail-safe: los Manifests son la autoridad. Object Lock / "GC desactivable" = add-on de tier compliance (Team), no default. HISTORIAL = PALANCA DE MONETIZACIÓN.

- **PRECIOS (arrancar con estos):** Free $0 perpetuo, devices ilimitados, SOLO self-hosted vault, tiempo real, historial 14 días. Pro $8/usuario-mes, devices ilimitados (VPS gratis = ventaja vs Tailscale), self-host o gestionado (add-on), historial 90 días, SIN GB regalados. Team-self-host $12/usuario-mes (bytes en el VPS del cliente, margen ~100%); Team-gestionado $18/usuario-mes con 1 TB pooled incluido, SSO/SAML, auditoría, roles, historial 1 año. Add-on storage gestionado $0.05/GB-mes (margen ~66%, ~59% a churn alto de builds). VPS en serve: gratis en Free/Pro; en Team 1 incluido + $1.50/VPS extra. Team desde 2 personas. Costo real del vault gestionado ~$0.0168/GB-mes. POSICIONAMIENTO: nunca "más barato que Dropbox" (Plus real $9.99/2 TB); competir por MODELO (cobrar por conectar), dev/self-host y cifrado del cliente. Correcciones verificadas: Tailscale Standard $8 (sin SSO) / Premium $20 mensual (con SSO), trata servidores como tagged resources ($1/mes tras 50); Convex Pro overage storage $0.03/GB.

- **MVP ("magia completa, todo lo demás recortado"):** construir el bucle vertical delgado entre **Mac + VPS Linux** (NO dos Macs — el VPS es más barato/desechable/scriptable, valida el caso de uso estrella y el login por código), con un Espacio de juguete de archivos NORMALES (no node_modules real, para no arrastrar el problema de binarios per-SO). DENTRO: motor de bloques con CDC real (FastCDC, no chunking fijo); índice local persistente por archivo + re-chunkeo solo del archivo tocado; Vault R2 con orden de commit estricto (todos los Blocks a R2 ANTES de la Revision en Convex); Coordinador Convex con Manifest que referencia Blocks por hash y pagina (doc tope 1 MiB); change feed bidireccional vía cliente Rust de Convex; supresión de eco en el watcher; conflictos = ambas versiones; login por código; daemon en foreground; **MAPEO DE SISTEMA DE ARCHIVOS DESDE EL DÍA 1** (ver abajo). RESERVAR el hueco de cifrado en el formato del Block (campo de algoritmo + nombre de objeto = hash del CIFRADO, NO del texto claro — cifrado convergente DESCARTADO, ver F6; data key por-Block envuelta por la Space key; dedup scope-cuenta) aunque el cifrado en runtime quede fuera. DENTRO también (F8/F10): re-scan completo contra el índice local al arrancar y al reconectar (lo exige el criterio de éxito c), con la Revision base guardada por path en el índice; el Device re-hashea todo Block que baja del Vault y verifica contra el hash esperado (integridad, F9); reservar el trait Vault abstracto (backend R2 implementado; self-hosted post-MVP). FUERA: cifrado en runtime, self-hosted vault, servicio instalado, billing, dashboard, GC/retención, escala, binarios per-SO. Secuencia: esqueleto de formato → motor+CDC+índice en 1 máquina (paso 0: dos procesos misma Mac) → Vault R2+orden de commit → Coordinador+Manifest paginado → media magia (2 procesos misma máquina) → feed bidireccional+supresión de eco → Mac+VPS Linux+corte de red real. Fake demo (video/mock estilo Dropbox) en paralelo desde semana 1. Criterio de éxito: (a) edito en A y aparece solo en B; (b) bidireccional sin loop de eco ni conflictos falsos; (c) corto la red, edito offline en ambos, reconecto, reconcilia sin perder datos; (d) cambio 1 línea de un archivo grande y SOLO suben los Blocks cambiados. DESCARTADO: sync unidireccional / copy-whole-file (calcifica mal el formato del Block).

- **MAPEO DE SISTEMA DE ARCHIVOS (metido en el MVP desde el día 1 por petición del usuario; es parte del formato = caro de cambiar):** capa de abstracción donde el motor habla un "FS canónico" y cada SO tiene su adaptador. Rutas canónicas OS-independientes en el Manifest (forward slash, relativas al root del Espacio, UTF-8 con normalización unicode definida). Case sensitivity: macOS insensible-preservando vs Linux sensible → colisión de solo-mayúsculas se trata como conflicto, no se sobre-escribe. Metadata portable: bit de ejecutable como mínimo; NO tocar finales de línea (sync byte-exacto). Diseñar el espacio para nombres ilegales/reservados de Windows aunque solo se valide al entrar Windows. Mac+Linux valida lo principal (mayúsculas + unicode + permisos).
